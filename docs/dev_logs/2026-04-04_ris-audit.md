## Key Findings

### Overall maturity: ~80% of RIS v1 design is implemented

All 4 core layers and the CLI have substantive working code. The system is not a scaffold.
A live operator can ingest documents, run precheck, query the knowledge store, and generate
reports today without any code changes.

### What works, confirmed operationally

- `research-health` returns 6 health check results with real metrics from run_log.jsonl
- `research-stats summary` shows 5 stored source documents, 0 derived claims, 1 precheck result
- Ingestion pipeline: extract → hard-stop → eval → chunk → store works end-to-end
- KnowledgeStore is SQLite-backed (NOT Chroma as some older docs implied)
- Chroma is a separate vector index (index.py), used alongside SQLite for semantic search
- FTS5 lexical search, RRF fusion, and optional cross-encoder reranking are all wired
- Precheck returns labelled GO/CAUTION/STOP with reason codes and evidence gap notes
- All 14 research-* CLI commands exist with real implementations

### Biggest gaps

1. **Cloud LLM providers are hard-blocked** — Gemini, DeepSeek, OpenAI, Anthropic all raise
   ValueError even if the guard env var is set. They are explicitly "v2 deliverables."

2. **ManualProvider gives hardcoded scores** — The default evaluation provider returns all
   dims=3, total=12 (just barely passes). Every document passes with ManualProvider. This
   means the evaluation gate provides no quality filtering without Ollama running.

3. **Twitter/X is entirely absent** — No adapter, no fetcher, not scheduled. Not a partial
   implementation — zero code exists.

4. **Discord alert sink not wired** — The existing `discord.py` notification module (already
   used by SimTrader gates) is NOT connected to RIS. RIS uses a generic WebhookSink / LogSink
   protocol instead.

5. **Two health checks are stubs** — `model_unavailable` and `rejection_audit_disagreement`
   always return GREEN regardless of actual system state.

### Surprises (spec vs implementation discrepancies)

| Item | Spec claim | Actual implementation |
|------|-----------|----------------------|
| Dedup threshold | 0.92 (feature docs) | 0.85 (code default) |
| Knowledge store backend | "Chroma polytool_brain external_knowledge partition" | SQLite for documents/claims; Chroma is separate index.py |
| Claim extractor | LLM-based (implied) | Entirely heuristic/regex, EXTRACTOR_ID="heuristic_v1" |
| Discord alerts | Implied wired | LogSink default; discord.py not connected to RIS |
| SSRN adapter | Listed in specs | No fetcher exists; AcademicAdapter only covers arXiv |

### Positive surprises

- The claim_relations table with SUPPORTS/CONTRADICTS/SUPERSEDES/EXTENDS is fully wired
- Contradiction tracking in precheck queries this table correctly
- Freshness decay is a real exponential decay with per-family half-lives from config JSON
- Eval replay infrastructure exists (replay.py, artifacts.py) for drift detection
- Precheck ledger (SQLite override audit trail) is more complete than expected
- Feature extraction (hedge density, numeric density, citation density, polymarket_relevance)
  runs deterministically before any LLM call — good architectural separation

## Open Questions and Recommendations

1. **Dedup threshold:** Should the feature doc be corrected to say 0.85, or should the code
   be changed to 0.92? Current state causes near-duplicates to be rejected more aggressively
   than documented. Low urgency but operator should be aware.

2. **Discord wiring:** The alert_sink protocol is designed to be pluggable. Wiring the existing
   `discord.py` module would take ~20 lines of code and give the operator real Discord alerts
   from health check failures. Recommended for v1.5.

3. **ManualProvider semantics:** The provider gives every document a passing score (12/20 total).
   This means the evaluation gate is effectively bypassed when Ollama is not running. Consider
   either (a) documenting this more prominently, or (b) adding a minimum word count or
   freshness check that ManualProvider does enforce.

4. **Two deferred health checks:** `model_unavailable` and `rejection_audit_disagreement` are
   GREEN stubs. These were marked deferred. The operator should not rely on health checks to
   detect Ollama outages — `research-health` will show GREEN even if Ollama is down.

5. **SSRN gap:** If the operator needs SSRN papers, the current workaround is to download the
   PDF manually and use `research-ingest --file paper.pdf`. The AcademicAdapter cannot fetch
   SSRN directly.
