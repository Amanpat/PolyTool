## What Was Built

### packages/research/scheduling/scheduler.py

Core scheduler module with:
- `JOB_REGISTRY` — list of 8 job descriptors (id, name, trigger_description, callable_name); no twitter_ingest
- 8 job callable functions (`_job_run_academic_ingestion`, `_job_run_reddit_polymarket`, etc.)
- `_JOB_FN_MAP` — dict mapping callable_name -> function object; used by both `run_job()` and tests
- `start_research_scheduler(_scheduler_factory, _job_runner)` — returns started scheduler; lazy APScheduler import
- `run_job(job_id) -> int` — public API for single-job invocation; 0 on success, 1 on unknown id or exception

### tools/cli/research_scheduler.py

CLI with three subcommands:
- `status [--json]` — lists all 8 registered jobs; no APScheduler import required
- `start [--dry-run]` — starts blocking scheduler loop or prints schedule and exits
- `run-job <job_id> [--json]` — invokes single job callable by id

### pyproject.toml

Added `[ris]` optional dep group with `apscheduler>=3.10.0,<4.0`. Added `packages.research.scheduling` to setuptools packages list.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed sys.modules patching fragility in test isolation**
- **Found during:** Task 3 (full suite regression run)
- **Issue:** Tests patching `sys.modules["tools.cli.research_acquire"]` worked in isolation but failed when the full test suite was run because the real module was already loaded by prior tests. Python's import machinery retrieves `tools.cli.research_acquire` via the attribute on the already-loaded `tools.cli` module, bypassing `sys.modules` lookup for the child module.
- **Fix:** Changed affected tests to patch `sched_mod._JOB_FN_MAP["_job_run_academic_ingestion"]` directly (and other affected job callables). This is reliable regardless of import order and does not depend on `sys.modules` state.
- **Files modified:** `tests/test_ris_scheduler.py`
- **Commit:** 6f151f1
