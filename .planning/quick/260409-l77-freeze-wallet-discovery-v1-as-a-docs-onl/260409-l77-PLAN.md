**5. ClickHouse Table Contracts**

Define three tables with exact column names, types, engines, and ordering keys.
Follow the project's ClickHouse conventions (ReplacingMergeTree for mutable state,
MergeTree for append-only). Reference the existing schema files under
`infra/clickhouse/initdb/` for naming and engine conventions.

Table: `watchlist`
- Engine: ReplacingMergeTree(updated_at)
- ORDER BY: (wallet_address)
- Primary identity: wallet_address (String)
- Columns:
  - wallet_address String
  - lifecycle_state Enum8('discovered'=1, 'queued'=2, 'scanned'=3, 'reviewed'=4, 'promoted'=5, 'watched'=6, 'stale'=7, 'retired'=8)
  - review_status Enum8('pending'=1, 'approved'=2, 'rejected'=3)
  - priority UInt8 DEFAULT 3
  - source String (e.g. 'loop_a', 'manual', 'loop_d')
  - reason String DEFAULT ''
  - last_scan_run_id Nullable(String)
  - last_scanned_at Nullable(DateTime)
  - last_activity_at Nullable(DateTime)
  - metadata_json String DEFAULT '{}'
  - updated_at DateTime DEFAULT now()
- Semantics: one current row per wallet (ReplacingMergeTree collapses on ORDER BY key, keeps latest updated_at). No auto-promotion; review_status must be manually set to 'approved' before lifecycle_state can advance to 'promoted'.

Table: `leaderboard_snapshots`
- Engine: MergeTree()
- ORDER BY: (snapshot_ts, order_by, time_period, category, proxy_wallet)
- Columns:
  - snapshot_ts DateTime
  - fetch_run_id String
  - order_by String (e.g. 'PNL', 'VOL')
  - time_period String (e.g. 'DAY', 'WEEK', 'MONTH', 'ALL')
  - category String (e.g. 'OVERALL', 'POLITICS', 'SPORTS', 'CRYPTO')
  - rank UInt32
  - proxy_wallet String
  - username String DEFAULT ''
  - pnl Float64
  - volume Float64
  - is_new UInt8 DEFAULT 0
  - raw_payload_json String DEFAULT '{}'
- Dedup key: (snapshot_ts, order_by, time_period, category, proxy_wallet) — enforced by ORDER BY; re-inserts with identical key are idempotent under MergeTree dedup.
- Semantics: append-only raw facts. Each fetch run appends a batch of rows. Historical snapshots are never mutated.

Table: `scan_queue`
- Engine: ReplacingMergeTree(updated_at)
- ORDER BY: (dedup_key)
- Columns:
  - queue_id String DEFAULT generateUUIDv4()
  - dedup_key String (wallet_address + source composite, e.g. 'loop_a:0xabc...')
  - wallet_address String
  - source String
  - source_ref String DEFAULT ''
  - priority UInt8 DEFAULT 3
  - queue_state Enum8('pending'=1, 'leased'=2, 'done'=3, 'failed'=4, 'dropped'=5)
  - available_at DateTime DEFAULT now()
  - leased_at Nullable(DateTime)
  - lease_expires_at Nullable(DateTime)
  - lease_owner Nullable(String)
  - attempt_count UInt8 DEFAULT 0
  - last_error Nullable(String)
  - created_at DateTime DEFAULT now()
  - updated_at DateTime DEFAULT now()
- Semantics: only one open item per dedup_key (ReplacingMergeTree keeps latest updated_at). A wallet can be re-queued after its current item reaches a terminal state (done, failed, dropped).

**6. Wallet Lifecycle State Machine**

Define the state machine with allowed transitions and invalid transitions:

States: discovered, queued, scanned, reviewed, promoted, watched, stale, retired

Allowed transitions (from -> to, trigger):
- discovered -> queued (scan queue picks up wallet)
- queued -> scanned (scan completes successfully)
- queued -> queued (scan fails, re-queued with incremented attempt_count)
- scanned -> reviewed (human operator reviews scan output)
- reviewed -> promoted (human approves; review_status must be 'approved')
- reviewed -> retired (human rejects; review_status set to 'rejected')
- promoted -> watched (operator activates for Loop B monitoring, future — not in v1)
- watched -> stale (no activity detected for configurable threshold, e.g. 30 days)
- stale -> queued (rescan triggered)
- stale -> retired (operator retires)
- any state -> retired (operator manual retirement)

Invalid transitions (explicitly document these):
- discovered -> promoted (skips mandatory scan and review)
- discovered -> watched (skips all gates)
- queued -> promoted (skips scan and review)
- scanned -> promoted (skips review — human gate is mandatory in v1)
- Any state -> discovered (discovered is an entry-only state, not reachable)

Note: In v1, the `watched` state exists in the schema but is not operationally
reachable because Loop B is out of scope. Wallets max out at `promoted` in v1.

**7. Deterministic Acceptance Tests**

Define these as test specifications (what to test, expected behavior). These are
contracts for the future implementation phase to write actual test code against.
No test code is written in this docs-only packet.

a) Leaderboard pagination fixture:
   - Given a mock leaderboard API returning 3 pages of 50 entries each
   - When fetch_leaderboard(max_pages=3) is called
   - Then 150 entries are returned, ordered by rank, with no duplicates
   - And all entries have non-empty proxy_wallet fields

b) Churn detection:
   - Given snapshot T-1 with wallets [A, B, C] and snapshot T with wallets [B, C, D]
   - When churn detection runs
   - Then wallet D is flagged as is_new=1
   - And wallet A is detected as dropped (present in T-1, absent in T)

c) Snapshot idempotency:
   - Given an identical leaderboard response fetched twice with the same snapshot_ts
   - When both are inserted into leaderboard_snapshots
   - Then the table contains exactly one set of rows (not duplicated) after OPTIMIZE

d) Queue dedup/lease behavior:
   - Given wallet 0xABC already in scan_queue with queue_state='pending'
   - When a second insert with the same dedup_key arrives
   - Then only one open item exists per dedup_key after OPTIMIZE
   - Given a pending item is leased (queue_state='leased', lease_expires_at=now()+300)
   - When lease_expires_at passes without completion
   - Then the item is available for re-lease (implementation detail: polling query filters available_at <= now())

e) Invalid lifecycle transition rejection:
   - Given a wallet in state 'discovered'
   - When an attempt is made to transition directly to 'promoted'
   - Then the transition is rejected (application-level validation, not ClickHouse constraint)
   - And an error is returned naming the invalid transition

f) Unified scan --quick no-LLM guarantee:
   - Given `python -m polytool scan <address> --quick` is invoked
   - Then no HTTP calls are made to any cloud LLM endpoint (Gemini, DeepSeek, OpenAI, Anthropic)
   - And the output contains MVF vector, existing detectors, PnL data
   - And the output does NOT contain LLM-generated hypothesis text

g) MVF fixture output shape:
   - Given a synthetic dossier JSON with known trade data
   - When MVF computation runs
   - Then the output contains all defined dimensions (initially 11; maker/taker ratio optional if data unavailable)
   - And each dimension is a float in a documented range
   - And the output includes a metadata block with wallet_address, computation_timestamp, and input_trade_count
