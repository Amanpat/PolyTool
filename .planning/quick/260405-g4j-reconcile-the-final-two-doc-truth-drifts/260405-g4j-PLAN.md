**Step 2 -- Rewrite the MCP section in RIS_OPERATOR_GUIDE.md (lines 599-628):**

Replace the entire "Claude Code MCP connection via n8n" subsection. The current content
falsely claims:
- HTTP transport at `http://localhost:{MCP_PORT}/mcp-server/http`
- Bearer token auth via `N8N_MCP_BEARER_TOKEN`
- Header Auth credential setup in n8n
- HTTP Request node pointing to `host.docker.internal:{MCP_PORT}/mcp-server/http`
- `MCP_PORT` env var usage

Replace with a truthful section that states:
- The MCP server uses **stdio transport only** (not HTTP).
- CLI invocation: `python -m polytool mcp` (optionally `--log-level DEBUG`).
- It is designed for Claude Desktop integration via stdio pipe, not for HTTP access
  from n8n or other services.
- n8n workflows do NOT use MCP. They use the docker-exec bridge pattern
  (`docker exec polytool-ris-scheduler python -m polytool ...`) to call CLI commands
  directly. This is documented in the n8n workflow templates section above.
- Remove references to `N8N_MCP_BEARER_TOKEN`, `MCP_PORT`, `host.docker.internal`,
  Header Auth credentials, and HTTP Request nodes for MCP.
- Add a note: "[PLANNED] HTTP transport for MCP is not implemented. If added in the
  future, this section will be updated."
- Remove `N8N_MCP_BEARER_TOKEN` from the environment variables table if present, or
  mark it as non-operative.

**Do NOT touch** the env vars table entry for N8N_MCP_BEARER_TOKEN in `.env.example` --
that is runtime config outside docs scope.
