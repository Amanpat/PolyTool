## Section 1: Module Inventory

Enumerate every Python package/module systematically:

1. **packages/polymarket/** — Read each .py file and subdirectory. For each:
   - Module path (e.g., `packages/polymarket/fees.py`)
   - One-line purpose (from docstring or first few lines)
   - Status: WORKING (has real logic + tests), STUBBED (skeleton/placeholder), DEAD (unused/orphaned)
   - Key classes/functions exported (grep for `class ` and `def ` at top level, plus `__all__` if present)
   - Subdirectories to cover: `crypto_pairs/`, `historical_import/`, `hypotheses/`, `market_selection/`, `notifications/`, `rag/`, `simtrader/` (including `simtrader/strategies/`, `simtrader/strategy/`, `simtrader/portfolio/`, `simtrader/shadow/`, `simtrader/execution/`)

2. **packages/research/** — Same treatment for all subpackages: `evaluation/`, `hypotheses/`, `ingestion/`, `integration/`, `monitoring/`, `scheduling/`, `synthesis/`

3. **polytool/** — The CLI entry package (`__main__.py`, `user_context.py`, `reports/`)

4. **tools/cli/** — Every CLI module file. Note which ones are active subcommands vs unused.

5. **tools/gates/**, **tools/guard/**, **tools/ops/**, **tools/smoke/**, **tools/setup/** — Utility scripts.

6. **Top-level scripts/** — Any standalone scripts.

7. **services/** — FastAPI or other service code.

Status determination heuristic:
- WORKING = has meaningful logic (>20 lines of non-import code) AND has at least one test file referencing it
- STUBBED = has skeleton/placeholder code, raises NotImplementedError, or has TODO markers as primary content
- DEAD = not imported by anything else, no test references, or explicitly deprecated

## Section 2: CLI Commands

Read `polytool/__main__.py` to find the argument parser and all registered subcommands. For each subcommand:
- Read the corresponding `tools/cli/<name>.py` file
- Extract: command name, brief description (from help text in argparse), subcommands if any
- Functional status: IMPLEMENTED (has real logic), STUB (placeholder), BROKEN (obvious issues like missing imports at module level)

Do NOT attempt to run any commands. Derive everything from source code reading.

## Section 3: Database State

ClickHouse:
- Grep for `clickhouse` (case-insensitive) across the codebase to find all table references, CREATE TABLE statements, and schema definitions
- Check `infra/` for any SQL schema files
- List every table name found with its apparent purpose

DuckDB:
- Grep for `duckdb` imports and usage across the codebase
- List every file that uses DuckDB and what it does

ChromaDB:
- Grep for `chroma` imports and collection names
- List collections and any partition/tag schemes found in code

## Section 4: External Integrations

Grep for:
- WebSocket connections (`websocket`, `ws://`, `wss://`)
- HTTP API clients (`requests.get`, `requests.post`, `urllib`, `httpx`, `aiohttp`)
- Known service patterns: `polymarket`, `gamma`, `coinbase`, `binance`, `clob`, `subgraph`, `thegraph`, `arxiv`, `reddit`, `youtube`, `discord`, `n8n`
- SDK imports: `py_clob_client`, `praw`, `yt_dlp`

For each integration found:
- Service name, module where it lives
- Protocol (REST, WebSocket, GraphQL, etc.)
- Status: TESTED (has tests), WRITTEN (code exists, no tests), STUB

## Section 5: Config and Environment

Environment variables:
- Grep for `os.environ`, `os.getenv`, `dotenv`, `.env` across all Python files
- List every env var name found with which file references it

Config files:
- List everything under `config/` with one-line description
- Note any config files elsewhere (pyproject.toml, pyrightconfig.json, docker-compose.yml, etc.)

## Section 6: Test Coverage

- List every file under `tests/` and map it to the module it tests
- Identify modules from Section 1 that have ZERO test files
- Note test count if easily discoverable (grep for `def test_` in each test file)

## Section 7: Known Duplication

Flag cases where:
- Two modules provide similar functionality (e.g., multiple HTTP client wrappers, multiple fee calculation paths)
- Same logic appears in multiple places (copy-paste patterns)
- Multiple config loading patterns exist

Use grep to search for function/class names that appear suspiciously similar across different files.

## Format

Use clean Markdown with tables where appropriate. Each section gets an H2 header. Subsections use H3. Status fields should be **bold** for easy scanning. Keep descriptions terse — one line per item where possible, expand only for notable findings.

## Approach

Work through the sections in order. For each section, use the Grep and Read tools systematically — do not guess or rely on CLAUDE.md claims. Read actual source files. Be brutally honest: if something looks half-built, say STUBBED. If something is clearly dead code, say DEAD. This audit exists to tell the truth.
