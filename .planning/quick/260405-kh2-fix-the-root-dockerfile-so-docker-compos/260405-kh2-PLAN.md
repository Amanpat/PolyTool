<objective>
Fix the root Dockerfile so `docker compose up -d --build` no longer fails on the ris-scheduler / default compose path.

Purpose: The builder stage runs `pip install ".[ris,mcp,simtrader,historical,historical-import,live]"` before the source tree exists, which causes setuptools to abort with:
  - warning: `/app/README.md` not found  (excluded by .dockerignore)
  - error: `package directory './polytool' does not exist`  (COPY comes later)

The fix: add two stub-creation RUN lines in the builder stage so setuptools can satisfy its metadata and package-directory requirements during the deps-only install. The real source is copied afterwards; the `--no-deps` reinstall then corrects entry points and metadata.

Output:
  - Dockerfile with stub creation before first pip install
  - Dev log
</objective>

```
# Stub README.md and package __init__ so setuptools can resolve metadata
# during the deps-only install. Real source is copied below and --no-deps
# reinstall fixes entry points. README.md is excluded from .dockerignore so

RUN echo "# PolyTool" > README.md \
    && mkdir -p polytool \
    && touch polytool/__init__.py
```

Insert this as a new layer between the COPY pyproject.toml line and the existing
`RUN --mount=type=cache,target=/root/.cache/pip` deps-install block.

DO NOT modify:
  - The BuildKit cache mount directives (`--mount=type=cache,...`)
  - The extras list `.[ris,mcp,simtrader,historical,historical-import,live]`
  - The `--no-deps` reinstall at the bottom of the builder stage
  - The runtime stage (FROM python:3.11-slim onward)
  - The COPY source lines (polytool/, packages/, tools/, services/)
  - .dockerignore (README.md remains excluded from build context — stub is created inline)

Final layer ordering in builder stage must be:
  1. apt-get install gcc libffi-dev
  2. COPY pyproject.toml ./
  3. RUN (new stub layer) — creates README.md + polytool/__init__.py
  4. RUN pip install --upgrade pip && pip install ".[extras]"  (deps cached here)
  5. COPY polytool/ packages/ tools/ services/  (real source)
  6. RUN pip install --no-deps ".[extras]"  (fixes entry points)

This preserves the BuildKit cache benefit: pyproject.toml changes invalidate
the deps layer; source-only changes do not.

## Root cause
The Dockerfile used a two-phase pip install pattern to cache deps separately
from source. Phase 1 (`pip install ".[extras]"`) ran before `COPY polytool/` and
before README.md entered the build context (.dockerignore excludes it). setuptools
required both to build package metadata, so the build aborted.

## Fix
Added a stub-creation RUN layer after `COPY pyproject.toml` and before the
deps-only install. The stub creates `README.md` inline and an empty
`polytool/__init__.py`, satisfying setuptools. The real source COPY overwrites
both stubs, and the subsequent `--no-deps` reinstall corrects entry points.

## Outcome
`docker compose up -d --build` passes on the default/ris-scheduler path.
BuildKit dep-cache layer preserved (cache miss only when pyproject.toml changes).

## Files changed
- Dockerfile (stub RUN layer added, all other layers unchanged)
